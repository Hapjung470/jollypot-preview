이 문서는 졸리팟 전체 기획 중 이번에 개발할 범위만 견적용으로 추려낸 정의서입니다. 5개 영역(UIUX 수정 · 레시피 퀄리티 업그레이드 · 냉장고 관리(신규) · 어드민 개편 · 앱 패키징(푸시·딥링크))을 각 프로토타입에 직접 연결해, 개발사가 링크를 눌러 세부 화면·상태까지 확인하고 견적할 수 있게 구성했습니다.

견적의 핵심 — 왜 일부만 개발하나

졸리팟 전체 기획을 한꺼번에 개발하지 않습니다. 전체 중 일부분만 린하게 먼저 개발하고, 실제 사용자와 핏(fit)을 맞춰본 뒤, 검증된 방향으로 추가 개발을 지속하는 방식입니다.

그래서 이번 견적 대상은 “돌려보며 검증할 최소 한 바퀴”에 필요한 화면·기능으로 한정했습니다. 전달드린 프로토타입 URL에는 전체 기획의 후반부 작업(스크랩북 SNS·요리모드·요리후기 등)이 함께 보일 수 있으나, 이번 개발 범위가 아닙니다. 혼선이 없도록 본 문서 하단에 ‘추후 개발’ 영역을 명확히 분리해 표기했습니다.

03 신규 기능 · 수동 중심 + 📷 사진 보조 이번 범위 · 신규

냉장고 관리 (신규) — 수동 중심 + 📷 사진 보조

수동 입력이 기본(spine)이고, 타이핑을 줄여주는 📷 사진 보조(‘보이는 것만’)가 더해졌습니다. 냉장고 ‘전체를 AI로 스캔’하는 방식이 아니라, 앞줄·포장식품만 빠르게 담는 보조입니다. 진입점은 홈(01번)과 연결됩니다.

⚑ 방식 — ‘전체 AI 스캔’은 폐기, ‘보이는 것만 보조’로 안착
2샷 촬영·스캔결과·스캔실패 화면, AI 비전 모델(Gemini/벤더) 채택 이슈는 제거. 대신 사진은 또렷이 보이는 것만 인식해 수동 추가 시트에 미리 채워주는 보조로 정리됐습니다(반찬통·안쪽은 처음부터 비대상).
냉장고 프로토타입fridge · v3-1-1 (수동+사진보조) 열기 →
빈 냉장고 재료 있음(양문형: 냉동/냉장) 📷 카메라 사진 결과(인식·불확실·0개)
빠른추가 칩 · 탭 1번 직접 입력 위치(냉동/냉장) 넣은 날 탭 → 수정·삭제 전체 비우기 📷 사진 보조: 빈 상태·추가 시트·더보기 3곳 진입
✓ 수동 + 사진 보조까지 화면 설계 완료 (프로토에 ‘개발 핵심사항’ 명세 내장)
수동: 담기(칩+직접입력+위치+넣은 날)·양문형 보관·수정/이동/삭제·전체 비우기. 넣은 날 자동기록·유통기한 미추적(의도). 사진 보조: 카메라(보이는 것만 선고지) → 결과를 별도 화면 없이 추가 시트에 미리 채움 → 확인 후 담기. 프로토 하단에 개발 핵심사항(모드 원칙·약속 범위·신뢰도·진입 위계)이 그대로 적혀 있어 개발사가 바로 참고 가능합니다.
📷 사진 보조 — 핵심 동작 규칙 (precision 우선)
고신뢰=실선·기본 포함(✕로 제외) / 저신뢰=점선 “이거 맞나요?”·기본 제외(‘네, 맞아요’ 눌러야 담김). 담기 카운트=확실+확인분만 — 보증 없는 추측은 저장 안 함. 모델 목표는 recall보다 precision. 위치는 추정→결과 행에서 토글 교정. ‘0개 인식’도 실패가 아님(직접 추가로 연결). 별도 ‘스캔 실패’ 화면 없음.
레시피 브릿지 — 냉장고 → 발견 루프
보관 재료 기준 “이 재료로 만들 수 있어요”(✓ 다 있어요 / 🛒 OO만 더). 추천은 확정 재료만 반영(불확실·미확인 제외).
⚑ 결정 필요 — 견적·구현 포인트
① 레시피 매칭 로직(보유 재료 ↔ 레시피 재료 정합 → 부족분 산출) — 냉장고의 실질 백엔드. ② ‘자주 넣는 재료’ 기본 사전(칩 세트) 정의. ③ 사진 인식 합격선 = precision 기준(애매하면 저신뢰로 흘려보내 사용자 확정). 모델은 레시피 추출용 Gemini Flash 재활용 전제 → 새 비전 벤더 결정 없음. ※ ‘빈 상태 스타일’은 인라인 빠른추가로 확정(2안 중 택1 완료).
범위 못박기: 수동 담기·양문형 관리·레시피 매칭 + ‘보이는 것만’ 사진 보조까지가 스코프. 냉장고 전체 자동 스캔은 범위 외.
정합 주의: 홈(01)의 냉장고 모듈 진입 문구가 아직 ‘냉장고 찍기(촬영)’ 톤이면, 현재 방식(수동+보이는 것 보조)에 맞춰 문구 정렬이 필요합니다.
📷 사진 보조 — 설계 근거 & 확장안 (영수증·바코드)
프로토에 반영된 냉장고 사진 보조의 근거(왜 ‘보이는 것만’인지·정확도 데이터)와, 다음 확장 후보인 영수증 OCR·바코드까지 별도 분석 문서에 정리돼 있습니다.

05 네이티브 확장 · 앱 패키징 개별 견적 · 재패키징 1회

앱 패키징 — 푸시 & 딥링크 (네이티브)

하이브리드 앱에 푸시 알림과 딥링크를 붙이는 네이티브 작업입니다. 앞의 4개(웹 범위)와 달리 스토어 재심사가 필요해 각각 개별 견적으로 분리합니다. 두 건 모두 네이티브를 건드리므로 재패키징·스토어 제출은 1회로 묶어 중복비용을 방지합니다.

5-A. 푸시 알림개별 견적
만드는 방식 — 인프라는 "얹기"(과견적 방지 핵심)
발송 인프라는 FCM/OneSignal 등 SaaS를 얹습니다(앱 푸시 발송은 무료). 개발사 산출물은 ①SDK 심기(네이티브) ②딥링크 라우팅(클릭→앱 화면) ③샵바이 회원·구매 데이터를 세그먼트로 연동 — 이 3가지로 한정합니다. 발송·이미지·예약·통계 UI는 SaaS 대시보드가 대체합니다.
딥링크 랜딩(P0)이미지 푸시(P0)예약 발송(P0) 세그먼트 T2(P0)수신동의·야간제한(법적·P0)발송 통계(SaaS대체) 개인화 토큰(고효과)자동화 1~2개·장바구니이탈/재입고(고효과) A/B 테스트 (2차) 복잡 세그먼트·RFM (2차) 리치버튼·인앱메시지 (2차)
운영 방식: 발송·예약·통계는 운영자가 SaaS 대시보드에서 직접 처리합니다(개발 최소). 어드민에 발송화면을 통합하는 방식은 이번 범위가 아닙니다.
5-B. 딥링크 (유니버설 + 디퍼드)개별 견적
구조 — 규칙표 한 번 → 모든 링크 자동 딥링크
상품마다 수작업이 아니라 URL 규칙표(route map)를 코드에 한 번 심습니다(예: /product/:id · /recipe/:id · /event/:id). 이후 그 패턴을 따르는 모든 표준 URL이 자동으로 딥링크로 작동합니다. 유니버설/앱링크(https 링크가 앱을 엶) 네이티브 등록 + 도메인 인증파일(assetlinks.json / apple-app-site-association) 호스팅, 디퍼드(미설치→설치 후 목적지 유지)는 SDK(Branch 등)로 처리합니다.
어드민에 붙는 것 — "딥링크 생성기" 화면 1개 (UTM 태깅 전용)
딥링크 전환은 URL 규칙표가 자동으로 처리하므로, 이 화면이 하는 일은 어느 매체로 내보냈는지 UTM 태그를 남기는 것뿐입니다. 어드민에 새 메뉴 [딥링크 생성기]를 붙입니다. 흐름 3단계: ①링크 붙여넣기(샵바이·졸리팟 URL을 그대로 붙여넣으면 대상 유형을 패턴으로 자동 인식) → ②캠페인 태그(마케팅 매체 utm_source · 자사 채널 utm_medium · 캠페인명 utm_campaign — 태그 종류는 ⚙ 태그 관리에서 운영자가 추가·삭제) → ③생성된 링크 + 복사. 운영자는 이 링크를 푸시·카톡·인스타·문자에 붙여넣습니다.
어드민 — 딥링크 생성기 열기 →
미설치 유저 폴백 — 상단 배너 한 줄
앱 미설치 유저가 딥링크를 열면 웹 페이지를 그대로 보여주고, 페이지 상단에 앱 배너 한 줄(44px)만 얹습니다. 콘텐츠를 가리지 않습니다. 구성: 앱 아이콘 + "졸리팟 앱에서 보기" + [열기] 버튼 + [✕] 닫기. [열기] → 설치자는 앱이 열리고, 미설치자는 스토어로. ✕ 닫으면 24시간 미노출. iOS는 OS 기본 Smart App Banner(apple-itunes-app 메타태그 1줄)로 대체 가능. 결제·주문서 화면에서는 표시하지 않습니다.
순서: 푸시 클릭→이동이 딥링크에 의존하므로 딥링크(규칙표·네이티브등록)를 먼저, 푸시가 이를 가져다 씁니다.
공통 전제 — 재패키징 1회 + 계정·인증서 이관
푸시·딥링크 모두 네이티브라 재패키징·스토어 제출은 1회로 묶습니다(양쪽 견적에 중복 계상 금지). 전제 조건: 긱스튜디오로부터 keystore(안드로이드 서명키)·APNs 인증서(iOS)·Firebase/스토어 콘솔 계정 이관 — 이것이 없으면 재패키징·앱링크 인증이 물리적으로 불가합니다.
받을 것: ①푸시 개별견적 ②딥링크 개별견적 — 각각 분리, 재패키징 비용은 한쪽에만.
